# Итоги экспериментов RNS v0.8 и v0.9

Этот файл фиксирует результаты экспериментов на Tesla T4 перед возвратом к базовой архитектуре v0.7. Он не предлагает переносить реализацию v0.8 или v0.9 в основную ветку целиком.

## v0.8 — эксперимент с компактным хранением

### Что было успешно

1. **Одна постоянная INT8-копия весов**
   - Хранение составило ровно `1 byte/weight`.
   - Для матрицы `768×768`: 589 824 байта.
   - Это соответствует native INT8, в 2 раза меньше FP16 и в 4 раза меньше четырёх residue planes v0.7.

2. **Четырёхканальная RNS-математика сохранилась**
   - В проверенных конфигурациях четыре канала воспроизводили установленный квантованный результат без ошибки INT32 reconstruction.

3. **Генерация residues из compact bytes технически возможна**
   - Не обязательно постоянно хранить четыре полные residue-матрицы весов в global memory.

4. **Fused Garner epilogue остался полезным**
   - Garner reconstruction, scales, bias и FP16 conversion можно выполнять без большого промежуточного INT64 output.

5. **Сильно улучшилась инфраструктура измерений**
   - Были отлажены CUDA Graph, NVTX, `cudaProfilerApi` capture, Nsight Systems summaries и Nsight Compute reports.

### Что было неуспешно

1. **Четыре RNS-канала выполнялись последовательно внутри одного thread**
   - Для `M=1, K=N=768` основной kernel запускался как `12 blocks × 64 threads`.
   - Его длительность была около 97 мкс против примерно 17 мкс у основного cuBLAS residue GEMM в v0.7.

2. **Был уничтожен параллелизм v0.7**
   - В v0.7 работа распараллеливалась по output tiles и каналам.
   - В v0.8 основная работа малого decode свелась к 768 threads.

3. **CUDA Graph показал реальную регрессию**
   - Без Python/launch шума v0.8 оказалась значительно медленнее v0.7, native INT8 и FP16.

4. **Не использовались Tensor Cores**
   - Реализация опиралась на scalar `__dp4a`, а не на SM75 INT8 IMMA.

5. **Генерация residues доминировала над dot product**
   - Распаковка байтов, modulo mapping, repack и integer control занимали большую часть instruction stream.

6. **Трёхканальный режим не стал безопасной общей заменой**
   - В одном тесте он совпал при `M=1`, но дал ненулевое отклонение при `M=16` и `M=128`.

7. **Полный FP16-input pipeline всё ещё состоял из многих коротких kernels**
   - Scale/reduction/quantization overhead не был устранён.

## v0.9 — эксперимент с warp-level parallelism

### Что было успешно

1. **Гипотеза о параллелизме для `M=1` подтвердилась**
   - Одна warp вычисляла один output column, а lanes делили K.
   - Decode core ускорился в несколько раз относительно v0.8.

2. **`direct4` оказался лучшей compact topology**
   - Вычислять четыре канала из одной загрузки compact weight выгоднее, чем схемы `2+2` и четыре независимые channel groups.

3. **Decode улучшился относительно v0.7**
   - `M=1, 768×768`: CUDA Graph p50 около 61,04 мкс против 79,49 мкс у v0.7.
   - `M=1, 4096×4096`: около 237,23 мкс против 395,79 мкс у v0.7 и 137,55 мкс у FP16.

4. **Компактное хранение `1 byte/weight` сохранилось**.

5. **Nsight опроверг цель «максимальная occupancy»**
   - `pair2` и `single1` имели более высокую occupancy, но были медленнее из-за повторных reads, partial writes и дополнительной integer work.

6. **Стало ясно, что GEMV и GEMM требуют разных kernels**
   - Warp-level compact kernel может быть полезен для autoregressive decode `M=1`.
   - Он не является заменой tiled GEMM для prefill.

### Что было неуспешно

1. **Архитектура была принципиально неправильной для `M≥16`**
   - Одинаковые weight residues генерировались заново для каждой строки M вместо повторного использования weight tile.

2. **При `M=128` производительность разрушилась**
   - CUDA Graph v0.9: примерно 1,074 мс.
   - FP16: примерно 43,25 мкс.
   - v0.7: примерно 119,87 мкс.
   - Этот результат доказывает, что warp-per-output реализация не была GEMM. Он не доказывает, что RNS по природе в 25 раз медленнее FP16.

3. **`pair2` и `single1` умножали memory traffic**
   - Compact weights читались повторно, а partial channel sums записывались в global memory.

4. **Split-K часто лишь переносил работу**
   - Ускорение partial main kernel компенсировалось workspace и вторым reduction/Garner kernel.

5. **Scalar DP4A остался неподходящим primitive для prefill**
   - Не было shared-memory tile reuse и SM75 Tensor Core mainloop.

6. **End-to-end pipeline сохранил launch overhead**
   - Non-Graph decode в значительной степени ограничивался preprocessing и последовательностью запусков.

7. **В первой упаковке v0.9 были инженерные ошибки**
   - Отсутствовала закрывающая brace namespace в CUDA-файле.
   - Неконтролируемое обновление setuptools конфликтовало с PyTorch 2.11.
   - Ошибки были исправлены в v0.9.1, но подтвердили необходимость замороженной базовой версии.

## Что сохранить после возврата к v0.7

- tiled/cuBLAS-подобную организацию GEMM v0.7;
- fused Garner/dequant/bias → FP16 epilogue;
- требование одной compact-копии весов как будущую цель;
- CUDA Graph, NVTX и полную Nsight-диагностику;
- идею `direct4` только как возможную отдельную ветку для `M=1`.

## Чего больше не делать

- не заменять tiled GEMM для большого M на warp-per-output;
- не оптимизировать число blocks или occupancy изолированно;
- не генерировать одинаковые weight residues отдельно для каждой строки M;
- не разделять каналы, если это дублирует compact weight reads;
- не оценивать близость к FP16 только по non-Graph timings;
- не менять одновременно storage, execution topology и epilogue.

## Рекомендуемый порядок новой работы

1. Заморозить и воспроизвести latency/Nsight базовой v0.7.
2. Добавить compact persistent storage, но временно материализовать residues перед неизменённым v0.7 GEMM; измерять materialization отдельно.
3. Заменить полную материализацию на tile-local residue generation с повторным использованием weight tile по нескольким строкам M.
4. Перенести выигравшую tiled topology на SM75 INT8 IMMA/Tensor Cores.
5. Оставить отдельный compact GEMV для `M=1` только при измеримом преимуществе.

На каждом этапе должна меняться только одна архитектурная переменная. Каждый результат необходимо сравнивать одновременно с FP16 и замороженной v0.7 под CUDA Graph и Nsight Systems.
