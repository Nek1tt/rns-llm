# Архитектура первого hybrid-прототипа

## Почему выбран per-channel split

Прототип не использует fallback целой строки или токена. В больших
Transformer выбросные значения часто встречаются во многих токенах, но
концентрируются в небольшом и относительно устойчивом наборе скрытых каналов.
Поэтому единицей дорогой обработки выбран **входной канал линейного слоя**.

Для слоя

\[
Y=XW^T+b
\]

индексы внутреннего измерения `K` делятся офлайн на множества `safe` и
`protected`:

\[
Y = X_s W_s^T + X_p W_p^T + b.
\]

Основная ветка `X_s W_s^T` выполняется native INT8. Только
`X_p W_p^T` отправляется в RNS или, в контрольном baseline, в FP16.

## Никакого априорного решения «продолжать»

Notebook сначала запускает статистический аудит на реальной модели. Для каждого
линейного слоя измеряются:

- доля строк, содержащих значения выше абсолютного порога;
- частота превышения порога по каждому каналу;
- концентрация энергии в top-1% каналов;
- устойчивость top-1% карты между calibration batches через Jaccard;
- channel-risk score, учитывающий активации и норму соответствующего столбца
  весов;
- доля защищённых каналов до и после padding до Tensor Core shapes;
- ожидаемая арифметическая граница hybrid относительно FP16;
- уменьшение ошибки native INT8 при q16-защите выбранных каналов.

По умолчанию статистический gate требует:

- защищать не более 3% входных каналов;
- mean top-1% Jaccard не ниже 0.5;
- уменьшить relative-L2 ошибку native INT8 минимум на 20%;
- иметь идеальную compute-оценку не хуже 0.9 времени FP16.

Не менее половины выбранных representative layers должны пройти gate. Иначе
решение записывается как `STOP`, а CUDA-benchmark hybrid-архитектуры не
запускается.

## CUDA dataflow

### Native main branch

1. FP32-активация собирается по `safe_indices`.
2. Выполняется per-row symmetric INT8 quantization.
3. Предварительно квантованные веса хранятся как `[K_safe, N] int8`.
4. Используется cuBLAS INT8 Tensor Core GEMM с INT32 accumulation.
5. Результат dequantize в FP32.

### Protected RNS branch

1. FP32-активация собирается по `protected_indices`.
2. Она квантуется логически в q8, q16 или q32.
3. Логические целые значения преобразуются в несколько 8-битных residue
   planes.
4. По каждому модулю запускается INT8 Tensor Core GEMM.
5. Канальные INT32-аккумуляторы восстанавливаются Garner-алгоритмом.
6. Для q32 используется U128 из двух 64-битных limbs: произведение модулей и
   результат не обрезаются до int64.
7. После dequantization protected contribution складывается с main branch.

## Почему q32 требует много каналов

Для logical bits `b` и длины dot product `K` worst-case bound равен

\[
B = K(2^{b-1}-1)^2.
\]

Произведение модулей должно удовлетворять

\[
\prod_i m_i > 2B.
\]

Поэтому q32 обычно требует около 9–10 восьмибитных residue channels. Этот режим
добавлен как требуемый научный baseline по точности и скорости, а не как
ожидаемый победитель.

## Что пока намеренно не оптимизировано

Это falsifiable layer-level prototype:

- `index_select` не слит с quantization;
- main и protected branch запускаются раздельно;
- merge выполняется отдельными PyTorch операциями;
- защищённая карта статична;
- нет end-to-end perplexity evaluation;
- нет перестановки каналов между соседними слоями;
- нет CUTLASS grouped/persistent kernel.

Если даже теоретически благоприятная статистика отсутствует, работа
останавливается до kernel engineering. Если статистика хорошая, но первый
нефьюзенный prototype проигрывает FP16, Nsight показывает, какая часть overhead
должна быть устранена.
